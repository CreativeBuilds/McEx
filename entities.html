1.2 Property pattern?

1.2.1 Base properties / Dynamic properties

  1. Base properties are set from the beginning, per entity type
  2. Dynamic properties are things like health/position/name/id
    1. These are the ones that get saved
  3. Same interface?
    1. Offer the ability to override base properties?
      1. Too powerful?
      2. Very customizable
      3. Final (non overridable) base properties?
        1. For things like entity type, should not change to prevent bad behavior

1.2.2 Property transitions?

  1. Need to handle changes in properties, send data to client/listeners
  2. We may need to handle transitions in multiple attributes at the same time
    1. Transaction?

1.2.3 Behaviors

  1. How do we act on events? (say input data from player)
  2. How do we act on property changes?
    1. Derived properties?

1.2.4 Thoughts

  1. Way to query things like position of all entities without iterating them all?
    1. Server listening to event bus, updating state?
      1. Could update central read-optimized ETS table?
      2. Easily queries by the entity processes themselves
  2. Avoiding infinite loops
    1. Should be easily runtime testable in most cases?
    2. Would like to avoid imposing constrains because of this
  3. This is starting to look like a kind of reactive programming?

1.2.5 Example events

  1. We get a move event from player client
    1. An event behavior gets triggered, updates the position
    2. Property behavior gets triggered by property update, send position event to event hub
  2. We get a move event from event hub (player entity)
    1. Event behavior gets triggered,
  3. An NPC moves once per tick
    1. Tick event behavior gets triggered, runs per-tick AI code, sets position property
    2. Property behavior gets triggered, send position to event hub
  4. A player punches another player

    This is an action that does not persist state in the same way as, say movement. (Is there any reason to handle it in a different way than things that do?) (Are there any actions like this we don't want to trigger a cooldown? Possibly from the server?) It would make sense to trigger a cooldown timer on actions like this, this also helps with anticheat. Cooldowns like this might be a very common action, do we want to take them into account in the implementation?

    1. An events gets triggered, updates the punch state? (timer?),
  5. A player walks off a block, he falls

1.2.6 Implementation

  1. Entity type represented by module
    1. Enables us to do more stuff at compile time in the future if important for performance
  2. Declare properties with an accumulated module attributes?
    1. Something like '@property :name'
    2. Convenient way to organize code, like a headline
  3. Code reusability provided by "behavior" modules with using macro
    1. Macro would defdelegate to the module itself
    2. A lot of common functionality could (probably) be implemented this way
      1. Health
      2. Movement
      3. Physics
    3. Ability to depend on other behaviors/properties?
      1. Not needed for now, can look into if manual ever becomes an issue
    4. Customized by properties(ideal? could be more dynamic. possibly only for things like max health?) or callbacks(maybe for more logicy things)
